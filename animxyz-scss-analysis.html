@animxyz/core v0.6.6 · scss source analysis · packages/core/src

One keyframe, a thousand dials

AnimXYZ's SCSS doesn't ship a library of animations — it ships a single parametric animation per mode whose every property is a CSS custom property, plus 1,441 tiny selectors that turn those dials. This is a walkthrough of how that engine works, where it creaks, and how much of it the platform can now do on its own.

The shape of the package

@animxyz/core is nine SCSS files that compile to one CSS file. Consumers almost never run the Sass themselves — the published main is dist/animxyz.css — but the sass field also exposes the source for people who want to customize the level maps or use the mixins. The module graph is clean, modern @use/@forward… with two exceptions, both visible below.

FileLinesRole
_variables.scss527Pure data. Level maps ($xyz-duration-levels…), the variable registry ($xyz-variables-map, 24 entries), and the utility schema ($xyz-utilities-map, 34 utilities declaring type, axis, target vars, levels, and an optional value transformer). All !default — this is the theming API.

02

The engine: dials, triggers, keyframes

Most animation libraries are combinatorial: fadeInUp, fadeInDown, zoomInLeft… one keyframe block per combination. AnimXYZ inverts this. There are exactly three @keyframes in the whole compiled output — one per mode (in out appear) — and every property inside them is a var() with an identity fallback: translate falls back to 0, scale to 1, opacity to 1. An unset dial contributes nothing; a set dial composes into the shared transform. Composition happens in the browser's custom-property resolution, not in the preprocessor.

Trace one element through the three layers:

markupauthor writes
<div class="xyz-in" xyz="fade up-100% duration-10">

The xyz attribute is a space-separated utility list — queried with [xyz~='…'], exactly like class. The class is the trigger; the attribute is pure configuration.

layer 1 · dials_utilities.scss → schema loop

Each utility selector sets custom properties — nothing else. Generated from $xyz-utilities-map: 34 utilities × 4 modes × ~10 levels ⇒ 1,441 selectors.

[xyz~='fade']         { --xyz-opacity: calc(1 - var(--xyz-opacity-default)); }
[xyz~='up-100%']      { --xyz-translate-y: calc(100% * -1); }
[xyz~='duration-10']  { --xyz-duration: 1s; }
layer 2 · trigger_core.scss → xyz-animation()

The mode class applies one animation whose timing slots all read the dial variables. Note the two animation names: the generated mode keyframes, plus a user-supplied keyframes hook (--xyz-keyframes) so custom animations ride the same timing.

.xyz-in {
  animation: $duration $ease calc(var(--xyz-delay-calc)) $iterate $direction both;
  animation-name: xyz-in-keyframes, var(--xyz-in-keyframes, var(--xyz-keyframes, …));
}
layer 3 · keyframes_core.scss:10–38

The single from frame composes every possible dial with identity fallbacks. Set none: nothing happens. Set any subset: they merge into one transform.

@keyframes xyz-in-keyframes {
  from {
    opacity: var(--xyz-in-opacity, var(--xyz-opacity, 1));
    transform: perspective(…)
      translate3d(…tx, 0, …ty, 0, …tz, 0)
      rotateX(…, 0) rotateY(…, 0) rotateZ(…, 0)
      scale3d(…, 1, …, 1, …, 1)
      skew(…, 0, …, 0);
  }
}

This is the architectural insight worth naming: Sass never composes anything. It only enumerates dial names and prints selectors. The composability that gives the library its identity is a pure CSS-custom-property mechanism — which matters a lot when we ask, later, whether Sass is still needed.

It also explains a subtle strength over the obvious "modern" alternative. You might reach for animation-composition: accumulate with one keyframe per utility — but animation-name lists don't merge across independent selectors, so [xyz~='fade'] and [xyz~='up'] could never each contribute an animation to the same element. Variables merge across selectors for free. The var-dial design isn't a preprocessor-era workaround; it's still the right architecture today.

03

The custom-property cascade

Every dial resolves through a fallback chain built by xyz-var() (_functions.scss:44). The chain encodes the precedence rules: mode-specific beats generic, element beats default, default beats identity. Two chain shapes exist:

Scoping: the [xyz] firewall

Custom properties inherit — normally a bug for a system like this, but AnimXYZ turns it into the nesting model, then contains it. Setting xyz="fade up" on a parent means nested children inherit those dials, which is how .xyz-nested elements animate from a single parent config. The containment half: every element carrying an xyz attribute resets all 96 dial variables (24 names × 4 mode prefixes) to initial (_core.scss:158) — so an inner xyz attribute is a clean override boundary rather than an accidental mix with ancestors. xyz="inherit …" flips the same 96 declarations to inherit for deliberate composition across the boundary.

It works, and it's clever — but it means every configured element carries ~96 declarations of plumbing, and it's a place where modern CSS (@property) can't simply swap in: registering these as non-inherited would kill the nesting feature. Inheritance here is load-bearing.

Stagger & the delay math

CSS has no sibling-index() in 2020, so AnimXYZ brute-forces one (_core.scss:54): a loop stamps :nth-child(1…20) → --xyz-index: 0…19 and :nth-last-child(1…20) → --xyz-index-rev onto every trigger and nested element. Forty rules, each carrying a large @extend-merged selector list — and a silent cliff: the 21st sibling has no index, falls back to 0, and animates in sync with the first.

05

Issues found

Ordered by severity. Everything tagged bug was reproduced against this working tree with Dart Sass 1.101.

bug · confirmed xyz-apply is broken for every input

The public mixin for baking utilities into your own selectors passes its arguments in the wrong order (_functions.scss:169). The signature is xyz-utility($name, $mode, $level) (:76), but the call sends ($name, $level, $mode) — so the level lands in the mode slot and vice versa, and the level lookup fails: 

@include xyz-apply('fade up');
Error: "all is not a valid level for the fade utility."

Two more faults hide behind the swap: mode detection uses substring matching, so thin and origin (both contain in) would resolve to mode in; and utility-name matching keeps the last map-order hit, working for flip-up vs up only because of the map's accidental ordering. The mixin appears in no docs, examples, or tests — which is presumably how it shipped broken. Fix the argument order and switch parsing to prefix/exact-token matching, or delete it.

bug · time-fused build.scss still uses @import

The one legacy-syntax file in the package is the build entry. Dart Sass warns on every build today and removes @import in 3.0 — at which point yarn build stops working. The fix is one line: @use 'src/animxyz' as *;. (The xyz-include-once guard in _internal.scss exists partly to defend against double-emission under @import semantics; with modules everywhere it becomes nearly vestigial.)

bug · dead code _fancy.scss is unreachable

Seven extra keyframe sets (expand-width, expand-height, expand-border-radius, filter-blur, filter-grayscale, box-shadow, stroke-draw) behind an xyz-fancy() mixin that nothing forwards, includes, or documents. Either wire it into the barrel as an opt-in module — SVG stroke-draw alone is a compelling utility — or remove it.

limitation Stagger silently caps at 20 siblings

$xyz-index-levels: 20 bounds the nth-child ladder. Child 21 gets --xyz-index: 0 and fires with child 1 — no warning, no wrap. Raising the number is a size trade: each extra level is 2 more rules dragging the full extend selector list. sibling-index() (§06) removes the cliff entirely where supported.

weight The [xyz] reset is ~200 shipped declarations

The inheritance firewall (§03) costs 96 declarations for [xyz] plus 96 more for [xyz~='inherit'], and the browser re-evaluates that block for every configured element. It's a necessary consequence of the design, but it's the single biggest fixed cost in the stylesheet, and worth remembering when weighing @property adoption — inheritance can't just be turned off (nesting depends on it).

stale workaround Internal *-calc variables ship to production

xyz-animation splits every delay expression into a --xyz-*-calc variable plus a calc(var(…)) consumer, flagged TODO against postcss-calc bug #77 (_functions.scss:190, 199). That bug was fixed upstream years ago. The workaround doubles the delay plumbing in the shipped CSS and leaks internals into the public variable namespace. Modern option: drop postcss-calc from the pipeline entirely — with var() everywhere, it has almost nothing left to safely fold.

fragility Cascade correctness by source order and !important

$xyz-modes: ['in', 'out', 'appear'] carries the comment "Appear must come last for specificity" — appear wins over in/out only because its rules are printed later. Meanwhile paused/none/absolute reach for !important to beat the trigger rules (_core.scss:130–142). Both are source-order load-bearing walls; user CSS with equal specificity can knock them over. Cascade layers give this a real structure (§06).

smell Data duplication in the schema

$xyz-duration-levels, $xyz-delay-levels, and $xyz-stagger-levels are three byte-identical 18-entry maps; the 24-entry variables map and 34-entry utilities map repeat near-identical per-axis stanzas. One shared time scale plus a small loop would more than halve _variables.scss — and since these maps are the public theming API, dedup means users override one scale, not three.

smell Small stuff

Hand-rolled xyz-str-split (_functions.scss:10) — string.split() has been in Dart Sass since 1.57. The @supports (transform: perspective(none)) probe (_core.scss:44) targets browsers that are now years obsolete. backface-visibility: visible sets the property's default value. And the bare xyz attribute is non-conforming HTML — harmless in practice, but data-xyz would validate and appease JSX tooling; worth considering (with an alias period) if there's ever a v1.0 break.

The library was designed in 2020 against 2020 browsers, yet its core mechanism — dials as custom properties, one parametric keyframe — is exactly how you'd design it today. Two modern features are deliberately absent from the table as replacements: animation-composition can't replace the var-dial design (independent attribute selectors can't each append an animation-name, as noted in §02), and non-inherited @property registration would break nested composition. Both are cases where the 2020 architecture already beat the 2026 feature.

Does it still need SCSS?

Not to run — only to be written. Nothing Sass does here survives to the browser: the runtime engine is entirely custom properties, calc(), and three keyframes, and the published artifact is already plain CSS. Sass performs exactly three jobs: (1) enumeration — expanding a 34-entry schema into 1,441 selectors; (2) theming — the !default level maps users override at build time; (3) a mixin API for Sass consumers (xyz-animation, xyz-utility, the level maps). Jobs it once did — nesting, math, vendor juggling — are gone or native.

A ~100-line Node script could emit the same stylesheet, and typed attr() will eventually shrink even the enumeration. But job 3 is a real public API and job 2 is the documented customization story, so dropping Sass today would break users to save a dependency that costs almost nothing. The pragmatic path: keep SCSS as the generator, but treat it as an implementation detail — fix the entry file, registered @property dials, cascade layers, sibling-index() behind a fallback — and spend the real innovation budget on the new capabilities column: @starting-style enter/exit and scroll-driven timelines are the features that would make AnimXYZ feel new in 2026, and both snap cleanly onto the dial architecture it already has.
